    Frontmatter:
      name: sg-use-tdd
      description: Use this when you want to enable or disable TDD mode — toggles .planning/USE-TDD marker on/off, or sets it explicitly with on/off argument.

    Include <language> block (same pattern as sg-health): detect user language, respond in kind; machine tokens (on/off/USE-TDD/paths) stay English.

    <objective>: Toggle TDD mode by managing .planning/USE-TDD marker file. Marker presence = TDD ON; absence = TDD OFF. Does NOT modify HANDOFF.md (this is a config toggle, not a workflow stage transition — per D-07).

    <execution_context>: Writes .planning/USE-TDD when enabling; deletes it when disabling. Reads nothing else.

    <process> must implement the three branches (per D-01, D-02, D-03):

    Step 1 — Parse $ARGUMENTS:
      Read $ARGUMENTS. Trim whitespace. Classify as: "on", "off", or empty/other (treat as no-arg toggle).

    Step 2 — Resolve current state:
      Check whether .planning/USE-TDD exists. Set TDD_ON=true if it exists, TDD_ON=false otherwise.

    Step 3 — Execute branch:
      BRANCH A ("on" argument):
        Run: mkdir -p .planning && touch .planning/USE-TDD
        Write marker content: a single human-readable line "TDD mode enabled" plus an ISO-8601 activation timestamp (per D-04). Use printf or echo with a heredoc-style write — no awk/sed.
        Output: "TDD mode: ON" (prose in user's language; "ON" stays English)

      BRANCH B ("off" argument):
        If .planning/USE-TDD exists, delete it: rm .planning/USE-TDD
        If it does not exist, do nothing (idempotent per D-03).
        Output: "TDD mode: OFF"

      BRANCH C (no-arg toggle):
        First output the current state: "TDD mode is currently: ON" or "TDD mode is currently: OFF"
        Then toggle: if TDD_ON=true → run BRANCH B logic; if TDD_ON=false → run BRANCH A logic.
        Output the new state after toggle: "TDD mode: ON" or "TDD mode: OFF"

    Idempotency guarantee (D-03): "on" when marker already exists = touch is a no-op but writes updated timestamp content. "off" when marker absent = rm skipped with no error.

    Marker file content (D-04, D-05):
      Before any write, run: mkdir -p .planning
      Content written on enable: two lines — "TDD mode enabled" and "Activated: <ISO timestamp>"
      Use: printf 'TDD mode enabled\nActivated: %s\n' "$(date -u +%Y-%m-%dT%H:%M:%SZ)" > .planning/USE-TDD
      (date -u +%Y-%m-%dT%H:%M:%SZ is POSIX-compatible on both macOS and Linux)

    Shell portability (CLAUDE.md): no grep -P. Argument parsing uses case or grep -E.

    <success_criteria>:
      1. Running the skill with no argument when .planning/USE-TDD is absent creates the marker and outputs "TDD mode: ON".
      2. Running the skill with no argument when .planning/USE-TDD is present deletes the marker and outputs "TDD mode: OFF".
      3. Running "on" twice leaves exactly one .planning/USE-TDD file and produces no error.
      4. Running "off" twice when the marker is absent produces no error and outputs "TDD mode: OFF".
      5. HANDOFF.md is not modified.
      6. .planning/ directory is created if it does not exist (mkdir -p).
  </action>

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| user argument → marker write path | $ARGUMENTS value (on/off/empty) controls whether a file is written to .planning/USE-TDD |
| skill process → filesystem | The process writes or deletes a file under .planning/; a bug could write outside this directory |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-46-01 | Tampering | .planning/USE-TDD write path | mitigate | Marker path is hardcoded as `.planning/USE-TDD` — not derived from $ARGUMENTS. No user-controlled path interpolation. |
| T-46-02 | Tampering | Repeated toggle calls (idempotency race) | mitigate | `touch` is idempotent on existing files; `rm -f` or `test -f && rm` is idempotent on absent files. No TOCTOU risk for a single-user local CLI. |
| T-46-03 | Denial of Service | mkdir -p .planning on read-only filesystem | accept | Local developer tool; read-only .planning/ is an operator misconfiguration, not an attack surface. mkdir exits non-zero and the error is surfaced to the user. |
| T-46-04 | Information Disclosure | Marker content (timestamp) | accept | Marker content is a human-readable one-liner with an ISO timestamp — no secrets, no PII. |
| T-46-SC | Tampering | npm/pip/cargo installs | accept | This plan installs no packages. No package legitimacy gate required. |
</threat_model>
