<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| local repo → Claude Code/Codex/Gemini hook runtime | hook runtime forks the configured command (`node ... .cjs`) with the agent's process privileges; config edits change which binary runs at hook-trigger time |

The phase touches only three local JSON config files. There is no user input crossing a boundary, no network call, no new package installed, no new external service. The surface area is limited to whatever the three command strings dispatch.

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-29-01 | Tampering | hook config command field | mitigate | Replace ONLY the two-token pair (`python3`→`node`, `.py`→`.cjs`) per D-05. Quoting and env-var expansion preserved byte-exact (D-06). Acceptance criteria in Task 2 lock exact target strings; Task 3 Tier 1 grep blocks any residual `python3`. Limits accidental introduction of unrelated commands during the edit. |
| T-29-02 | Denial of Service | hook target script missing | mitigate | Pre-flight guard (Task 1, D-12) and re-check immediately before commit (Task 3) confirm all 4 `.cjs` files exist and respond to fixture input. Prevents the silent-failure landmine documented in 29-CONTEXT.md §"위험·landmine" where a missing `.cjs` would cause "command not found" on every hook trigger. |
| T-29-03 | Tampering | JSON syntax corruption | mitigate | Task 3 Tier 2 parses each file with `node -e "JSON.parse(...)"`. Edits use surgical token replacement (no JSON reformat), so structural corruption is unlikely; the parse step is a belt-and-suspenders guard. |
| T-29-04 | Information Disclosure | n/a — no secrets, no PII | accept | The three config files contain no credentials, no PII, no secrets. The command strings reference `${CLAUDE_PLUGIN_ROOT}` and `$GEMINI_PROJECT_DIR` (path env vars) and literal hook script names — public information already in the repo. No mitigation required. |
| T-29-05 | Repudiation | unauthorized config change | accept | Single atomic commit (D-09) with explicit subject and co-author trailer creates an unambiguous audit trail. Git history is the system of record. No additional logging needed for a 3-file surgical config edit. |
| T-29-06 | Elevation of Privilege | n/a — no new entry points | accept | The phase does not add new hook events, new external services, new network calls, or new auth surfaces. Hook event types (PreToolUse, Stop, SubagentStop, SessionEnd, BeforeTool) remain identical to pre-Phase-29 state. The only change is which interpreter binary forks. No privilege escalation surface. |
| T-29-SC | Tampering | npm/pip/cargo installs | accept | No package installs in this phase. No `package.json`, `requirements.txt`, or `Cargo.toml` modified. The legitimacy audit table requirement is not triggered. |

The threat surface is intentionally minimal — this is a config-only token swap. Honest assessment: most of the STRIDE categories do not apply because no new capabilities are introduced. The two real risks (T-29-01 byte-exact correctness, T-29-02 missing target script) are mitigated by the pre-flight guard, surgical edit constraints, and 4-tier verification gate before commit.
