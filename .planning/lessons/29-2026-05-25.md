### Draft sg-rules

```yaml
# .claude/sg-rule.warn-json-escape-verify.local.md
name: warn-json-escape-verify
enabled: true
event: file
conditions:
  - field: file_path
    operator: regex_match
    pattern: '\.planning/phases/.+/.+-PLAN\.md$'
  - field: new_string
    operator: regex_match
    pattern: 'grep\s+-F[^"]*"[^\\]*"'
action: warn
---
PLAN.md `<verify><automated>` block contains `grep -F` with unescaped double quotes. If the target file is JSON, the file bytes contain `\\"` escapes — the pattern will false-negative against a correct implementation.

Fix: either (a) escape the quotes in the grep pattern to match file bytes, or (b) verify via `node -e "JSON.parse(...)"` and compare values semantically.
```

```yaml
# .claude/sg-rule.warn-verify-gate-head-assumption.local.md
name: warn-verify-gate-head-assumption
enabled: true
event: file
conditions:
  - field: file_path
    operator: regex_match
    pattern: '\.planning/phases/.+/.+-PLAN\.md$'
  - field: new_string
    operator: regex_match
    pattern: 'git\s+log\s+-1\s+(--pretty|--name-only)'
action: warn
---
PLAN.md verify gate uses `git log -1 ...` which assumes the feat commit stays at HEAD. After-execute SUMMARY + HANDOFF docs commits will push HEAD past the feat commit, making the gate exit 1 on a fully-correct shipped state — cannot be replayed for forensics.

Fix: anchor by commit hash (`git log $FEAT_SHA -1 ...`) or use `git rev-list --grep` with the subject pattern.
```
