# Technology Stack ‚Äî v1.3 Codex Platform Support

**Project:** super-gsd v1.3
**Researched:** 2026-05-21
**Scope:** Stack additions required to support OpenAI Codex CLI users. Existing Claude Code stack is NOT re-researched here ‚Äî only delta for Codex is documented.

---

## What Codex Platform Support Requires

The goal is to let Codex users run super-gsd workflows without a separate install path. The minimal approach is: one AGENTS.md that teaches Codex the sg- workflow, plus `.agents/skills/` (the canonical project-scoped path) containing adapted skill files. No new Python, no new hooks, no forking of the 13 existing commands.

---

## Finding 1: AGENTS.md ‚Äî File Format and Location

**Status:** HIGH confidence (official docs)
**Source:** https://developers.openai.com/codex/guides/agents-md

### Exact specification

- **File:** Plain Markdown. No frontmatter. Any section headers. No YAML.
- **Project-root location:** `AGENTS.md` at repo root ‚Äî this is what Codex reads first for project-wide guidance.
- **Override file:** `AGENTS.override.md` in same directory ‚Äî checked before `AGENTS.md`; use for temporary overrides, not for the primary file.
- **Nested scoping:** Codex walks down from repo root to the current working directory, reading one file per directory. Closer files win.
- **Global location:** `~/.codex/AGENTS.md` ‚Äî user-level defaults loaded before project files.
- **Size limit:** 32 KiB per file (`project_doc_max_bytes`). Discovery stops when combined size hits the limit.
- **Context injection:** Each discovered file becomes a `user`-role message prefixed with `# AGENTS.md instructions for <directory>`. Injected near top of conversation, root-to-leaf order, before user prompt. Rebuilds on every session start.
- **Empty files:** Skipped entirely.

### Existing AGENTS.md problem

The repo already has an `AGENTS.md` at root (line 1 in the file read above), but it was auto-generated by GSD's `gsd-new-project` with stale content referencing "Codex-plugins-official/superpowers" and `.Codex-plugin/plugin.json` ‚Äî these are placeholder strings, not real Codex paths. This file must be **replaced**, not appended to.

### What the new AGENTS.md must contain

1. What super-gsd is and its 3-stage workflow (GSD plan ‚Üí Superpowers execute ‚Üí sg-retro)
2. The 13 sg- command names and when to invoke each (user types `$sg-start`, not `/sg-start`)
3. `.planning/` state file locations (STATE.md, HANDOFF.md) so Codex knows project context
4. Reference to `.agents/skills/` for the adapted skill files

## Finding 2: Skills Directory ‚Äî Canonical Location

**Status:** HIGH confidence (official docs)
**Source:** https://developers.openai.com/codex/skills

### Directory structure

```
repo-root/
‚îî‚îÄ‚îÄ .agents/
    ‚îî‚îÄ‚îÄ skills/
        ‚îî‚îÄ‚îÄ sg-retro/
            ‚îî‚îÄ‚îÄ SKILL.md
```

**This is the canonical project-scoped path.** Codex scans `.agents/skills/` in every directory from CWD up to repo root. Conflicting sources mention `.codex/skills/` as a valid project path, but the official Agent Skills page specifies `.agents/skills/` as the primary project convention. Use `.agents/skills/` to be certain.

Global (cross-project) skills go to `~/.codex/skills/` or `~/.agents/skills/`. For super-gsd, only the project-scoped `.agents/skills/` is needed ‚Äî we want skills version-controlled in the repo.

### SKILL.md format

```markdown
---
name: skill-name
description: When/why to trigger this skill (used for implicit matching)
---

[Skill instructions follow as plain markdown]
```

YAML frontmatter with `name` and `description` is **required**. The `description` field drives implicit invocation ‚Äî Codex uses it to auto-select the skill when task descriptions match. This is the same YAML-frontmatter pattern already used in `commands/sg-*.md` (e.g., `sg-start.md` has `name:`, `description:`, `argument-hint:`).

### Optional metadata file

`agents/openai.yaml` inside the skill directory enables UI configuration and explicit-only invocation:

```yaml
policy:
  allow_implicit_invocation: false  # prevent auto-triggering
```

For sg- commands, `allow_implicit_invocation: false` is recommended ‚Äî these commands have precise entry conditions that should not be guessed by Codex.

### Invocation in Codex

Users invoke skills with `$skill-name` (dollar sign, not slash). In Claude Code, commands use `/sg-start`. In Codex, they use `$sg-start`. This is the primary UX difference users need to know.

## Finding 3: Hooks System ‚Äî Critical Delta from Claude Code

**Status:** HIGH confidence (official docs)
**Source:** https://developers.openai.com/codex/hooks, https://github.com/openai/codex/issues/21753

### Supported hook events in Codex

| Event | Claude Code | Codex | Notes |
|-------|------------|-------|-------|
| `Stop` | YES | YES | Identical semantics |
| `PreToolUse` | YES | YES | Same JSON input/output spec |
| `PostToolUse` | YES | YES | Available |
| `UserPromptSubmit` | YES | YES | Available |
| `SessionStart` | YES | YES | Codex: filters on `source` (startup/resume/clear) |
| `SubagentStop` | YES | **NO** | Missing in Codex. Issue #21753 open. No ETA. |
| `PermissionRequest` | NO | YES | Codex-only |

**SubagentStop is the critical gap.** The existing `hooks/hooks.json` registers `SubagentStop` pointing to `stop_hook.py`. This hook fires in Claude Code when Superpowers (a subagent) completes a review. In Codex, `SubagentStop` does not exist ‚Äî the hook will silently do nothing or error.

### Codex hooks.json location

```
repo-root/
‚îî‚îÄ‚îÄ .codex/
    ‚îî‚îÄ‚îÄ hooks.json     ‚Üê project-scoped hooks (requires user trust before running)
```

Format is **identical** to Claude Code's `hooks.json`. The JSON schema is the same ‚Äî `hooks.EventName[].hooks[].type/command/timeout`. The `CLAUDE_PLUGIN_ROOT` env variable used in the existing hook commands will not be set in a Codex session. Replace with a relative path or `$CODEX_HOME`-based path.

### Output field support difference

| Output field | Claude Code Stop | Codex Stop |
|---|---|---|
| `continue` | YES | YES |
| `stopReason` | YES | YES |
| `systemMessage` | YES | YES |
| `suppressOutput` | YES | YES |

For PreToolUse, Codex supports `permissionDecision: "deny|allow"` plus `additionalContext`. Claude Code's PreToolUse supports `decision: block` with `reason`. These are syntactically different. The existing `rule_runner.py` outputs Claude Code's PreToolUse format ‚Äî it needs a Codex compatibility check.

## Finding 4: Minimal Viable Approach ‚Äî No Fork

**Verdict:** Do NOT fork the 13 `commands/sg-*.md` files. The correct approach is:

### Layer 1: Rewrite `AGENTS.md` at repo root

Replace the current stale AGENTS.md with one that:
- Explains the workflow (GSD ‚Üí Superpowers ‚Üí sg-retro) in plain English
- Lists all 13 sg- commands with brief descriptions
- Tells Codex to invoke them as `$sg-start`, `$sg-plan`, etc.
- Points to `.planning/STATE.md` and `.planning/HANDOFF.md` for state

This single file covers ~90% of Codex users who just want to understand what commands exist.

### Layer 2: `.agents/skills/` with adapted SKILL.md files

Copy the most critical commands (sg-start, sg-plan, sg-execute, sg-review, sg-retro, sg-status) into `.agents/skills/` with proper SKILL.md frontmatter. These are light wrappers that reference the original `.planning/` files ‚Äî they do not duplicate the full command logic.

The sg-retro skill is already in `skills/sg-retro/SKILL.md` with proper frontmatter. It is directly reusable ‚Äî copy or symlink to `.agents/skills/sg-retro/SKILL.md`.

### Layer 3: `.codex/hooks.json` without `SubagentStop`

Create `.codex/hooks.json` with only `Stop` and `PreToolUse` entries. Omit `SubagentStop` entirely since Codex does not support it. The `Stop` hook covers the main workflow transition detection.

**For `rule_runner.py` in Codex context:** The `CLAUDE_PLUGIN_ROOT` env var will not exist. Use a path relative to the hook script's location or detect the env at runtime:

```python
PLUGIN_ROOT = os.environ.get('CLAUDE_PLUGIN_ROOT') or os.path.dirname(os.path.dirname(os.path.abspath(__file__)))
```

This one-line fix makes `rule_runner.py` and `stop_hook.py` work in both Claude Code and Codex without branching.

### What NOT to do

- Do NOT create a parallel `codex-commands/` directory with 13 duplicate files
- Do NOT write a new hook runner for Codex ‚Äî reuse the Python hooks as-is with the path fix above
- Do NOT introduce npm packages or a JS layer ‚Äî the existing Python 3 stdlib hooks work in Codex (same OS)
- Do NOT create new state files ‚Äî `.planning/STATE.md` and `.planning/HANDOFF.md` are platform-agnostic Markdown
- Do NOT implement a SubagentStop workaround ‚Äî it's an open issue upstream, the fallback is that users manually call `$sg-review` to trigger the retro step

## Finding 5: npm / CLI Packages Specific to Codex Integration

**Verdict:** None needed.

- `@openai/codex` ‚Äî the Codex CLI itself (`npm i -g @openai/codex`, requires Node 22+). This is the user's responsibility to install before using super-gsd on Codex. super-gsd does not need to depend on it.
- `@openai/codex-sdk` ‚Äî programmatic SDK, not relevant for a skill/hook plugin
- No Codex-specific plugin registry exists comparable to Claude Code's marketplace. Distribution is via repo (`.agents/skills/`) or the GitHub clone pattern.
