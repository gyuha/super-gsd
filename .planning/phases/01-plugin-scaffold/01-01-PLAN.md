| Boundary | Description |
|----------|-------------|
| GitHub repo → end-user install | A user runs `/plugin marketplace add gyuha/super-gsd` and trusts that whatever this repo serves (marketplace.json + plugin.json) is what they think they're installing. |
| Identity strings → marketplace listing | Fields like `name`, `author`, `owner`, `description` shown to users via `/plugin list` originate entirely from these static JSON files. |

## STRIDE Threat Register

Phase 1 ships only static JSON and Markdown — no executable code, no network calls, no hooks, no commands. The threat surface is minimal. The realistic threats are supply-chain / identity-confusion issues, not code execution issues.

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-01-01 | Spoofing | `plugin.json` author / repository fields | mitigate | `author` is locked to literal string `"gyuha"` (D-03); `repository` is resolved from `git remote get-url origin` rather than hand-typed, so it cannot drift from the actual git origin during execute time. Acceptance criteria check both values explicitly. |
| T-01-02 | Tampering | Cross-file identity drift between plugin.json and marketplace.json | mitigate | Task 2 acceptance criterion explicitly cross-references `plugins[0].name` against `.claude-plugin/plugin.json`'s `name` field at verify time, blocking accidental rename in one file but not the other. |
| T-01-03 | Information Disclosure | Author email accidentally embedded in author/owner field | mitigate | D-03 and D-06 explicitly forbid email; acceptance criteria in Tasks 1 and 2 check `author` and `owner` are plain strings (`"gyuha"`), and Task 2 asserts no `@` character appears anywhere in marketplace.json (sanity guard). |
| T-01-04 | Repudiation | Missing license / changelog | mitigate | LICENSE (MIT) and CHANGELOG.md are required artifacts; acceptance criteria confirm presence of `MIT License` heading and `[0.0.1]` version entry. |
| T-01-SC | Supply chain (package installs) | npm/pip/cargo install steps | n/a | Phase 1 installs NO external packages — only writes static JSON / text files. The RESEARCH.md package legitimacy gate is not triggered because no install task exists. If a future phase adds dependencies, that phase's plan must reinstate this gate. |
| T-01-05 | Denial of Service | Marketplace registry pointing at non-existent plugin source | accept | `source: "."` resolves to the same repo, which by definition exists at install time. No realistic DoS vector at Phase 1 surface. |
| T-01-06 | Elevation of Privilege | Plugin executing arbitrary code on install | n/a | Phase 1 ships no executable artifacts (no commands, no hooks, no scripts). Vector is structurally absent until Phase 2/3. |
</threat_model>
